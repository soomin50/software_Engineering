# Project I: 프로젝트 정의서 (시스템 정의서)

## (1) 프로젝트 명칭
조직 내 안전한 자료 관리를 위한 파일 공유 시스템 개발 프로젝트

## (2) 개발 시스템 명칭
미니 드라이브 (Mini Drive)

## (3) 개발 시스템에 대한 간략한 설명
미니 드라이브는 조직 내부에서 사용하는 파일을 한곳에서 저장, 관리, 공유할 수 있도록 지원하는 웹 기반 시스템이다.

기존에는 이메일, 메신저, 개인 클라우드 등을 통해 자료를 주고받으면서 파일 버전이 섞이거나, 필요한 자료를 다시 찾는 데 시간이 걸리고, 외부 공유 과정에서 보안 문제가 발생할 가능성이 있었다. 이러한 불편을 줄이기 위해, 사용자가 보다 간단하고 안전하게 파일을 관리할 수 있는 경량 클라우드 파일 공유 시스템을 구축하고자 한다.

## (4) 개발 시스템에서 제공하는 핵심 기능
1. **파일 및 폴더 관리 기능**
   * 다양한 형식의 업무 파일을 업로드하고 다운로드할 수 있다.
   * 폴더 생성, 이름 변경, 이동, 삭제 기능을 제공하며 삭제된 파일은 일정 기간 복구할 수 있다.

2. **파일 공유 및 권한 설정 기능**
   * 특정 사용자에게 파일을 공유하고, 조회·수정·댓글 권한을 구분하여 부여할 수 있다.
   * 외부 사용자와 공유할 경우 만료 기간이 포함된 링크를 생성할 수 있다.

3. **파일 검색 기능**
   * 파일명뿐 아니라 업로드 날짜, 업로더, 파일 종류 등의 조건을 활용해 원하는 파일을 찾을 수 있다.

4. **버전 관리 기능**
   * 동일 파일이 수정될 경우 이전 버전을 확인할 수 있으며, 필요 시 특정 버전으로 복원하거나 다운로드할 수 있다.

5. **계정 및 보안 관리 기능**
   * 이메일과 비밀번호를 이용한 로그인 기능을 제공한다.
   * 데이터 전송 시 보안을 적용하고, 관리자는 사용자 계정과 저장 용량을 관리할 수 있다.

## (5) 기존 유사 소프트웨어 조사 및 분석

* **Google Drive**
  * 파일 저장과 공유, 권한 설정, 버전 관리 등 본 시스템과 유사한 핵심 기능을 제공한다.
  * 다양한 협업 기능이 강점이지만, 단순한 파일 공유 목적만 놓고 보면 기능이 많아 다소 복잡하게 느껴질 수 있다.

* **OneDrive**
  * 조직 단위 사용 환경과의 연계성이 좋고, 파일 이력 관리 기능도 제공한다.
  * 기업 환경에 적합한 구조를 참고할 수 있다는 장점이 있다.

* **Dropbox**
  * 파일 동기화와 직관적인 사용 경험 측면에서 강점을 가진다.
  * 단순하고 빠른 파일 관리 경험이라는 점에서 참고할 가치가 있다.

## (6) 프로세스 모델 선정
미니 드라이브는 이미 유사 서비스가 존재하는 비교적 명확한 요구사항을 가지지만, 파일 보안과 안정성, 사용자 증가에 따른 성능 문제, 향후 기능 확장 가능성까지 함께 고려해야 한다.

따라서 본 프로젝트는 반복적인 개선과 위험 관리를 함께 수행할 수 있는 방식이 적합하다. 초기에는 파일 업로드·다운로드와 같은 핵심 기능을 우선 구현하고, 이후 반복적으로 검색 성능, 권한 관리, 버전 관리, 확장 기능 등을 보완하는 방향으로 개발하는 것이 적절하다고 판단된다.

# Project II: 대상 시스템 품질 요소 추정하기

요구사항 명세서와 시스템 정의 내용을 바탕으로, 클라우드 파일 공유 시스템인 **미니 드라이브(Mini Drive)** 에서 중요하게 고려해야 할 품질 요소를 이해관계자별로 정리하였다.

## (1) 사용자가 원하는 기대치는 무엇인가? (외부 품질 중심)
일반 사용자는 시스템 내부 구조보다는 실제 사용하면서 느끼는 편리함과 안정성을 중요하게 생각한다.

* **사용성 (Usability):** 사용자는 복잡한 학습 없이도 파일 업로드, 폴더 정리, 검색, 공유 기능을 쉽게 사용할 수 있기를 기대한다. 즉, 직관적인 화면 구성과 단순한 사용 절차가 중요하다.
* **신뢰성 (Reliability):** 파일이 정상적으로 저장되고, 다운로드 과정에서 오류가 없으며, 삭제 복구나 버전 복원 기능도 문제없이 동작해야 한다.
* **성능 효율성 (Performance Efficiency):** 검색 결과는 일반적인 조건에서 **2초 이내**에 표시되고, 대용량 파일도 과도한 지연 없이 업로드 및 다운로드되어야 한다. 이를 통해 사용자는 업무 중 불필요한 대기 시간을 최소화할 수 있다.
* **보안성 (Security):** 사용자는 자신이 올린 파일이 허가된 사람에게만 공유되고, 외부 링크를 통한 접근도 안전하게 통제되기를 원한다.

## (2) 개발자 혹은 유지보수 담당자가 원하는 기대치는 무엇인가? (내적 품질 중심)
개발자와 유지보수 담당자는 시스템이 장기적으로 관리 가능한 구조인지에 더 큰 관심을 가진다.

* **유지보수성 (Maintainability):** 파일 관리, 검색, 권한 제어, 버전 관리 기능이 서로 명확히 나뉘어 있어야 이후 수정이나 기능 추가가 수월하다.
* **검증 가능성 (Verifiability):** 권한 설정, 공유 링크, 복구 기능, 버전 복원 등의 동작이 테스트하기 쉬운 형태로 설계되어야 한다.
* **테스트가능성 (Testability):** 파일 업로드/다운로드, 권한 설정, 링크 만료, 버전 복원과 같은 핵심 기능에 대해 테스트 케이스를 쉽게 작성하고 반복 실행할 수 있어야 한다. 또한 기능 수정 이후에도 동일한 품질을 확인할 수 있도록 자동화된 검증이 가능해야 한다.
* **가독성 (Readability):** 여러 개발자가 함께 작업하거나 담당자가 바뀌더라도 코드를 이해하기 쉽도록 일관된 구조와 명확한 표현이 필요하다.
* **확장성 (Scalability):** 향후 공동 편집이나 AI 기반 검색 같은 추가 기능이 들어와도 기존 구조를 크게 흔들지 않고 확장할 수 있어야 한다.

## (3) 개발 조직의 관리자(스폰서)가 원하는 기대치는 무엇인가? (운영 및 관리 중심)
프로젝트 관리자나 스폰서는 시스템이 조직에 실질적인 가치를 주면서도 안정적으로 운영되기를 기대한다.

* **보안성 (Security):** 조직 내부 문서와 자료를 다루는 만큼, 무단 접근이나 정보 유출을 방지할 수 있는 수준의 보안이 반드시 보장되어야 한다.
* **가용성 (Availability):** 다수의 사용자가 동시에 접속하더라도 시스템이 자주 멈추지 않고 지속적으로 서비스를 제공해야 한다.
* **생산성 (Productivity):** 제한된 일정과 자원 안에서 핵심 기능을 우선적으로 구현하여 실질적인 결과를 낼 수 있어야 한다.
* **비용 효율성 (Cost Effectiveness):** 불필요하게 복잡한 기능까지 무리해서 구현하기보다는, 필요한 기능에 집중하여 적절한 비용으로 개발되는 것이 바람직하다.

> **종합 요약:** 미니 드라이브는 사용자에게는 쉽고 안정적인 파일 공유 환경을 제공해야 하며, 개발자에게는 수정과 검증이 쉬운 구조를 갖추어야 한다. 또한 관리자는 보안성과 가용성을 바탕으로 효율적으로 운영 가능한 시스템이 되기를 기대한다.
