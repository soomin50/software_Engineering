# 1. 서론

## 1.1 문서의 목적 및 범위

본 문서는 조직 내부에서 사용할 수 있는 클라우드 파일 공유 시스템인 **Mini Drive** 프로젝트를 수행하기 위한 프로젝트 관리 계획서이다.

본 문서는 Mini Drive 프로젝트의 목표, 개발 절차, 일정, 역할, 품질 관리 방법, 개발 환경, 산출물 관리 방법, 리스크 관리 방안을 정의하여 프로젝트를 체계적으로 수행하는 것을 목적으로 한다.

본 문서의 범위는 다음과 같다.

1. Mini Drive 프로젝트의 개요 및 목표 정의
2. 주요 기능과 개발 범위 정의
3. 개발 절차 모형 및 작업 분해 구조 정의
4. 개발 일정 및 역할 정의
5. 품질 관리와 변경 사항 관리 방법 정의
6. 산출물 관리 및 리스크 대응 방안 정의

## 1.2 프로젝트 개요

### 1.2.1 프로젝트 정의

Mini Drive는 조직 내부 사용자를 위한 웹 기반 클라우드 파일 공유 시스템이다.

현재 조직 내 파일은 이메일, 메신저, 개인 클라우드 등에 흩어져 공유되는 경우가 많다. 이러한 방식은 최신 파일 확인을 어렵게 만들고, 파일 검색 시간을 증가시키며, 외부 공유 시 보안 문제가 발생할 가능성이 있다.

Mini Drive는 파일을 중앙 저장소에서 관리하고, 사용자가 파일을 업로드, 다운로드, 공유, 검색할 수 있도록 지원한다. 또한 폴더 관리, 권한 설정, 외부 링크 공유, 버전 관리 기능을 제공하여 조직 내 파일 관리의 효율성과 보안성을 높이는 것을 목표로 한다.

### 1.2.2 주요 기능 설명

| ID | 기능명 | 설명 |
|---|---|---|
| F01 | 파일 업로드 | 사용자가 파일을 시스템에 업로드한다. |
| F02 | 파일 다운로드 | 사용자가 저장된 파일을 다운로드한다. |
| F03 | 폴더 관리 | 폴더 생성, 이름 변경, 삭제, 파일 이동을 지원한다. |
| F04 | 파일 공유 | 조직 내부 사용자에게 파일을 공유한다. |
| F05 | 공유 권한 설정 | 보기, 수정, 댓글 등의 권한을 설정한다. |
| F06 | 외부 링크 공유 | 외부 협력 업체와 공유할 수 있는 링크를 생성한다. |
| F07 | 파일 검색 | 파일 이름, 업로드 날짜, 사용자, 파일 유형으로 검색한다. |
| F08 | 버전 관리 | 파일의 이전 버전을 확인, 다운로드, 복원한다. |
| F09 | 사용자 계정 관리 | 사용자 로그인, 로그아웃, 계정 관리를 지원한다. |
| F10 | 보안 관리 | 인증, 권한 기반 접근 제어, 링크 만료 기간 설정을 적용한다. |

## 1.3 프로젝트 목표

본 프로젝트의 목표는 조직 내부 사용자가 파일을 안전하고 편리하게 저장, 공유, 검색할 수 있는 Mini Drive 시스템을 개발하는 것이다.

세부 목표는 다음과 같다.

1. 조직 내 파일을 중앙에서 관리할 수 있도록 한다.
2. 사용자가 파일을 쉽게 업로드하고 다운로드할 수 있도록 한다.
3. 폴더 기능을 통해 파일을 체계적으로 분류할 수 있도록 한다.
4. 사용자 간 파일 공유 기능을 제공한다.
5. 외부 사용자와 공유할 수 있는 링크 기능을 제공한다.
6. 파일 검색 기능을 통해 필요한 자료를 빠르게 찾을 수 있도록 한다.
7. 파일 수정 시 이전 버전을 확인하고 복원할 수 있도록 한다.
8. 로그인과 권한 관리를 통해 파일 접근 보안을 강화한다.
9. GitHub를 활용하여 문서와 소스코드를 체계적으로 관리한다.

# 2. 개발 계획

## 2.1 개발 절차 모형

본 프로젝트는 **애자일 방식**을 적용한다.

Mini Drive는 파일 업로드, 다운로드, 공유, 검색, 버전 관리, 권한 설정 등 여러 기능으로 구성되어 있다. 이러한 기능들은 한 번에 완성하기보다 우선순위에 따라 나누어 구현하고, 각 단계마다 테스트와 리뷰를 통해 개선하는 방식이 적합하다.

Scrum 방식을 선택한 이유는 다음과 같다.

1. 기능을 작은 단위로 나누어 순차적으로 개발할 수 있다.
2. 개발 중 요구사항 변경이 발생해도 유연하게 대응할 수 있다.
3. GitHub Issue 또는 Pull Request를 통해 리뷰와 수정 과정을 관리하기 쉽다.
4. 각 기능을 구현한 뒤 바로 테스트하고 개선할 수 있다.
5. 과제 요구사항인 리뷰 수행 및 리뷰 반영 과정을 개발 절차에 포함하기 적합하다.

## 4.2 변경 사항 관리 방법

프로젝트 진행 중 요구사항이나 문서 내용이 변경될 경우 다음 절차에 따라 관리한다.

1. 변경 사항을 GitHub Issue 또는 commit message에 기록한다.
2. 변경 전 내용과 변경 후 내용을 구분하여 작성한다.
3. 변경 사유를 명확히 정리한다.
4. 일정, 기능, 산출물에 미치는 영향을 확인한다.
5. 필요한 경우 README와 관련 문서를 함께 수정한다.
6. 수정 후 GitHub에 다시 업로드한다.

변경 사항 예시는 다음과 같다.

| 변경 항목 | 변경 전 | 변경 후 | 사유 |
|---|---|---|---|
| 프로젝트 주제 | 랩실 홈페이지 | Mini Drive | 과제 주제에 맞게 수정 |
| 문서 구조 | 간단한 계획서 형식 | 샘플 문서 기반 계획서 형식 | 첨부된 프로젝트 관리 계획서 형식 반영 |
| 산출물 위치 | README에 위치 미기재 | README에 각 과제 결과물 위치 명시 | 과제 제출 조건 충족 |
| 리뷰 반영 | 리뷰 기록 없음 | Issue 리뷰 내용 반영 | 과제 요구사항 반영 |

산출물은 GitHub Repository에서 관리한다.

### 4.3.1 파일 명명 규칙

| 산출물 종류 | 파일명 예시 |

### 4.3.2 저장 위치

| 산출물 | 저장 위치 |
|---|---|

### 4.3.3 버전 관리 방법

1. 모든 문서와 소스코드는 GitHub에 업로드한다.
2. 문서 변경 시 commit message에 수정 내용을 작성한다.
3. 리뷰를 반영한 경우 commit message에 리뷰 반영 사실을 기록한다.
4. README에는 각 과제 결과물의 위치를 명시한다.

예시 commit message는 다음과 같다.

```text
docs: Mini Drive 프로젝트 관리 계획서 작성
docs: README에 과제 결과물 위치 추가
docs: issue 리뷰 반영하여 프로젝트 주제 수정
```
