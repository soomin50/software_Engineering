# 1. 서론

## 1.1 문서의 목적 및 범위

본 문서는 조직 내부에서 사용할 수 있는 웹 기반 클라우드 파일 공유 시스템인 **Mini Drive**의 요구사항을 정의하기 위한 문서이다.

Mini Drive는 조직 구성원이 업무 파일을 중앙 저장소에 업로드하고, 필요한 파일을 다운로드, 검색, 공유, 관리할 수 있도록 지원하는 시스템이다. 기존에는 파일이 이메일, 메신저, 개인 클라우드 등에 분산되어 관리되기 때문에 최신 파일 확인이 어렵고, 파일 검색 시간이 길어지며, 외부 공유 과정에서 보안 문제가 발생할 수 있다.

본 문서는 이러한 문제를 해결하기 위해 Mini Drive가 제공해야 할 기능적 요구사항, 비기능적 요구사항, 인터페이스 요구사항, 데이터 요구사항, 개발 및 운영 요구사항을 정의하는 것을 목적으로 한다.

본 문서의 범위는 다음과 같다.

1. Mini Drive 프로젝트의 목적과 대상 시스템 정의
2. 파일 업로드, 다운로드, 폴더 관리, 공유, 검색, 버전 관리 기능 요구사항 정의
3. 사용자 계정, 권한 관리, 관리자 기능 요구사항 정의
4. 성능, 보안, 사용성, 신뢰성 등 비기능적 요구사항 정의
5. 사용자 화면, 외부 공유 링크, 서버 및 저장소 연동 등 인터페이스 요구사항 정의
6. GitHub 리뷰 및 리뷰 반영을 고려한 문서 관리 요구사항 정의

## 1.2 대상 시스템 개요

### 1.2.1 프로젝트 정의

Mini Drive는 조직 내부 사용자를 위한 웹 기반 클라우드 파일 저장 및 공유 시스템이다.

사용자는 웹 브라우저를 통해 시스템에 접속하여 파일을 업로드하고 다운로드할 수 있으며, 폴더를 생성하여 파일을 체계적으로 관리할 수 있다. 또한 내부 사용자에게 파일을 공유하거나 외부 협력 업체에게 공유 링크를 전달할 수 있다.

시스템은 사용자 계정 관리, 파일 검색, 권한 설정, 버전 관리, 삭제 파일 복구, 관리자 기능을 제공하여 조직 내 파일 관리의 효율성과 보안성을 높이는 것을 목표로 한다.

---

### 1.2.2 프로젝트 배경

조직 내 업무 파일은 다양한 방식으로 공유된다. 대표적으로 이메일 첨부, 메신저 파일 전송, 개인 클라우드 링크 공유 등이 사용된다. 그러나 이러한 방식은 다음과 같은 문제를 가진다.

1. 파일이 여러 경로에 분산되어 최신 버전을 확인하기 어렵다.
2. 파일 이름이나 저장 위치가 통일되지 않아 필요한 자료를 찾는 데 시간이 오래 걸린다.
3. 외부 사용자에게 파일을 공유할 때 접근 권한과 만료 기간을 관리하기 어렵다.
4. 파일이 수정되었을 때 이전 버전을 확인하거나 복원하기 어렵다.
5. 관리자 관점에서 사용자별 저장 공간과 공유 상태를 파악하기 어렵다.

Mini Drive는 이러한 문제를 해결하기 위해 파일을 중앙 저장소에서 관리하고, 파일 공유와 검색, 권한 제어를 하나의 시스템 안에서 처리하도록 설계한다.

| ID | 기능명 | 설명 |
|---|---|---|
| F01 | 파일 업로드 | 사용자가 업무 파일을 Mini Drive에 업로드한다. |
| F02 | 파일 다운로드 | 사용자가 저장된 파일을 자신의 PC로 다운로드한다. |
| F03 | 폴더 관리 | 사용자가 폴더를 생성, 수정, 삭제하고 파일을 이동한다. |
| F04 | 파일 공유 | 사용자가 파일 또는 폴더를 내부 사용자와 공유한다. |
| F05 | 공유 권한 설정 | 보기, 수정, 댓글 등의 권한을 설정한다. |
| F06 | 외부 링크 공유 | 외부 협력 업체와 공유할 수 있는 링크를 생성한다. |
| F07 | 파일 검색 | 파일 이름, 업로드 날짜, 파일 유형, 업로드 사용자 기준으로 검색한다. |
| F08 | 버전 관리 | 파일 수정 이력을 저장하고 이전 버전을 확인, 다운로드, 복원한다. |
| F09 | 사용자 계정 관리 | 로그인, 로그아웃, 사용자 계정 생성 및 삭제를 지원한다. |
| F10 | 관리자 기능 | 사용자 목록, 저장 공간 사용량, 공유 상태를 관리한다. |
| F11 | 보안 관리 | 인증, 권한 기반 접근 제어, 링크 만료 기간, HTTPS 통신을 적용한다. |
| F12 | 삭제 파일 복구 | 삭제된 파일을 일정 기간 동안 보관하고 복구할 수 있도록 한다. |

# 2. 요구사항

## 2.1 기능적 요구사항

기능적 요구사항은 Mini Drive 시스템이 사용자에게 제공해야 하는 기능을 정의한다.

---

### 2.1.1 사용자 계정 기능

| 분류 | 요구사항 | 우선순위 |
|---|---|---|
| FR-001 | 사용자는 이메일과 비밀번호를 입력하여 로그인할 수 있어야 한다. | 상 |
| FR-002 | 사용자는 로그인 후 시스템의 파일 관리 기능을 사용할 수 있어야 한다. | 상 |
| FR-003 | 사용자는 시스템 사용 후 로그아웃할 수 있어야 한다. | 상 |
| FR-004 | 시스템은 로그인하지 않은 사용자가 파일 목록, 업로드, 다운로드, 공유 기능에 접근하지 못하도록 해야 한다. | 상 |
| FR-005 | 관리자는 신규 사용자 계정을 생성할 수 있어야 한다. | 상 |
| FR-006 | 관리자는 기존 사용자 계정을 삭제할 수 있어야 한다. | 중 |
| FR-007 | 관리자는 사용자 계정의 활성화 또는 비활성화 상태를 변경할 수 있어야 한다. | 중 |
| FR-008 | 시스템은 사용자 정보를 기반으로 파일 접근 권한을 구분해야 한다. | 상 |

## 5.2 요구사항 작성 기준

본 요구사항 정의서는 다음 기준을 따랐다.

1. 요구사항은 기능적 요구사항, 비기능적 요구사항, 인터페이스 요구사항, 데이터 요구사항, 개발 및 운영 요구사항으로 구분하였다.
2. 각 요구사항은 “시스템은 ~해야 한다”, “사용자는 ~할 수 있어야 한다”와 같이 검토 가능한 문장으로 작성하였다.
3. 각 요구사항에는 고유 식별자를 부여하였다.
4. 각 요구사항에는 우선순위를 상, 중, 하로 구분하여 표시하였다.
5. Mini Drive 프로젝트 관리 계획서에서 정의한 주요 기능을 기준으로 요구사항을 세분화하였다.
6. GitHub Issue 또는 Pull Request를 통한 리뷰 수행과 리뷰 반영 절차를 요구사항에 포함하였다.
